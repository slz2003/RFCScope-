## Issue 1 — Wrong index used for `crc[1]` field-presence conditions

**Errata title:** `crc[1]` `failInfo` / `certifiedKeyPair` presence conditions incorrectly depend on `crc[0].status`

**Concise explanation:**  
Appendix C.4 profiles an `ip` message containing `CertRepMessage.response` with two `CertResponse` entries (`crc[0]`, `crc[1]`). For `crc[0]`, it correctly says `failInfo` is present iff `crc[0].status.status == "rejection"` and `certifiedKeyPair` is present iff `crc[0].status.status ∈ {"accepted","grantedWithMods"}`.  
But for `crc[1]`, the profile incorrectly ties `crc[1].status.failInfo` and `crc[1].certifiedKeyPair` to **`crc[0].status.status`** instead of `crc[1].status.status`, creating a cross-entry dependency that contradicts the per-`CertResponse` semantics in the main body.

**Justification:**  inconsistency  
- Section 5.3.4 defines per-entry semantics: in *each* `CertResponse`, presence of `failInfo` vs certificate/`CertifiedKeyPair` depends on that response’s own status (“Only one of … can be present in each CertResponse (depending on the status)”).  
- The Appendix C.4 text for `crc[1]` makes mixed outcomes (e.g., `crc[0]=accepted`, `crc[1]=rejection`, or vice versa) impossible to encode without violating either the profile or the base semantics, causing interoperability failures in multi-cert responses.

**Minimal patch suggestion:**  
In Appendix C.4 (Initialization Response — `ip`), change the two `crc[1]` conditions:

- **OLD:**  
  - `crc[1].status.failInfo` present iff `crc[0].status.status == "rejection"`  
  - `crc[1].certifiedKeyPair` present iff `crc[0].status.status == "accepted" or "grantedWithMods"`

- **NEW:**  
  - `crc[1].status.failInfo` present iff `crc[1].status.status == "rejection"`  
  - `crc[1].certifiedKeyPair` present iff `crc[1].status.status == "accepted" or "grantedWithMods"`

**Impacted dimensions:** Deontic, Structural, Causal, Boundary, Scope

**Experts that flagged it:** Scope, Causal, Deontic, Structural, Boundary
